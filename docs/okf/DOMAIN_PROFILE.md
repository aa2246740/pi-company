# pi-company OKF Domain Profile

Profile id: `works.pi-company.project-company`

Initial profile version: `0.1.0`

## Purpose

This profile defines the semantic concept types used by pi-company project OKF bundles. Bare OKF only guarantees parseable Markdown/YAML; this profile defines what pi-company concepts mean.

## Shared metadata fields

Recommended concept fields:

```yaml
type: RoleProfile
title: "Tester role profile"
status: active
authority: project-canonical
owner: lead
content_origin: system-seeded
source_refs: []
timestamp: "2026-06-26T00:00:00Z"
last_verified_at: null
review_due_at: null
expires_at: null
sensitivity: project-internal
strategy_mode: descriptive
influence:
  enabled: false
```

## Authority values

- `system-seeded`: generated by pi-company templates at initialization.
- `project-canonical`: accepted project knowledge for descriptive context.
- `role-authored`: produced by the responsible role.
- `derived`: produced from runtime evidence or deterministic projection.
- `imported-unverified`: imported from outside this project and not trusted for action.

## Status values

- `active`: eligible for context retrieval.
- `draft`: not used in default context.
- `superseded`: retained for history; not used by default.
- `stale`: may be shown with warning; not used for action.
- `archived`: retained but ignored by default.

## Concept types

### `BundleManifest`

Declares bundle ownership, purpose, profile compatibility, sensitivity, and influence boundary.

### `ProjectMission`

Describes what the local company is trying to accomplish for this project. It is descriptive context, not a completion signal.

### `SourceOfTruthInventory`

Documents where facts, runtime state, generated projections, and safety policies live.

### `ProjectConstraint`

Describes project-specific constraints, e.g. ŌĆ£browser-first RN validationŌĆØ or ŌĆ£do not mutate production dataŌĆØ. Constraints are read as context; executable restrictions must still live in code or company config.

### `RoleProfile`

Defines a roleŌĆÖs mission, temperament, decision bias, evidence policy, stop policy, handoff policy, forbidden actions, and rubric references.

### `EvaluationRubric`

Defines evidence criteria and scoring language for review, test, product acceptance, design quality, implementation quality, or other evaluator tasks.

### `ProjectPolicy`

Describes governance policy for humans and agents. Policy concepts may explain execution rules, but body text cannot grant permissions.

### `SprintContract`

Defines the agreed unit of work for one sprint: scope, done criteria, non-goals, evidence required, evaluator roles, linked issue/PR ids, and activation state.

### `EvaluationFinding`

Records tester, reviewer, PM, or system findings against a sprint contract or PR head. It may summarize evidence, but PR gate events remain authoritative for merge readiness.

### `StructuredHandoff`

Transfers context between sessions/roles. It must include current owner, runtime ids, evidence, blockers, next owner, and freshness.

### `VerificationTrace`

Records commands, browser/DOM checks, screenshots, or simulator evidence used during verification. It is an audit trail, not a pass by itself.

## Influence boundary

Every concept that can shape agent behavior must explicitly remain descriptive in v0.2:

```yaml
strategy_mode: descriptive
influence:
  enabled: false
```

Consumers must ignore any concept that tries to set `influence.enabled: true` until a later version explicitly implements an audited influence model.
