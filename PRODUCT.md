# Product

## Register

brand

## Users

Pi users, Codex users, and open source agent-tool builders who want visible multi-agent project work without giving up live terminal takeover. They are usually evaluating whether pi-company is trustworthy enough to run on a real local codebase.

## Product Purpose

pi-company is a Pi-native local runtime for one-project agent companies. It coordinates visible Pi agents through local issues, mailboxes, human steering, git worktrees, local PRs, review, test, and gated merges. Success means a user can see each agent, interrupt any agent, and still keep project state coherent.

## Brand Personality

Pragmatic, transparent, technical. The product should feel like a working developer tool with an auditable process, not like a generic AI automation pitch.

## Anti-references

Do not look like a generic SaaS AI landing page with vague productivity claims, floating gradient blobs, glass cards, or decorative agent diagrams. Do not hide the terminal-native nature of the product. Do not imply a headless orchestrator when the core promise is visible, steerable Pi panes.

## Design Principles

1. Show the operating model: make agents, mailboxes, worktrees, gates, and human steering visible.
2. Prefer evidence over claims: use concrete workflow artifacts, commands, and states.
3. Keep the human in the loop: make takeover and steering feel central, not incidental.
4. Treat roles as context boundaries: avoid fake corporate hierarchy and vague team metaphors.
5. Make local-first trust obvious: show that state is project-local and recoverable from events.

## Accessibility & Inclusion

Target WCAG AA for the public demo surface. Support keyboard use, visible focus states, readable contrast, responsive layouts, and reduced-motion preferences. Avoid color-only status communication for gates, roles, and agent states.
